AI Reflection Log

A critical evaluation of how AI tools were used throughout this project — what they produced, what was changed, and what limitations were discovered.

Overview

Throughout the development of WeaselCodes, three AI tools were used in different capacities. Claude (Anthropic) was the primary tool, used for content writing, concept explanation, and general development guidance. GitHub Copilot was used within Visual Studio Code for inline code suggestions and debugging assistance. Gemini (Google) was used specifically for generating the AI mascot image used across the site. Every AI output was reviewed, tested, and modified before being incorporated into the project. The use cases documented below reflect real interactions that shaped specific parts of this application.

Use case log

Specific examples of AI assistance used during this project, including the prompt, the output, and what was changed.

Use case 1

Project structure
Claude Sonnet 4.6

Prompt used:
I want to create proper directories for my web design project. What is the industry standard folder structure for an HTML CSS JS project?

What the AI produced:
Claude suggested a standard project structure with a single css/style.css file for all page styles, a single js/main.js for all JavaScript, and an assets/img/ folder for images. The suggestion was one stylesheet for all styling and one script file for all behaviour.

Changes made:
The base structure was adopted but it was extended beyond what the AI suggested. Rather than a single stylesheet, separate CSS files were created per page — cv.css for the CV page and quiz.css for the quiz page, alongside the shared style.css. Similarly, quiz.js was kept separate from main.js to isolate quiz logic. A data/ folder was also added to store questions.json, which the AI did not suggest. These decisions were made to follow the separation of concerns principle and make each file's purpose immediately obvious to anyone reading the project.

Use case 2

Explaining concepts
Claude Sonnet 4.6

Can you explain how AJAX works in JavaScript? I need to understand it well enough to implement it myself for loading quiz questions from a local JSON file and fetching data from a public API.
What the AI produced:
Claude explained that AJAX allows the browser to fetch data from a server without reloading the page, and walked through how XMLHttpRequest works step by step: creating the object, opening a GET request, setting request headers, defining the onload callback, and sending the request. It also explained how to parse a JSON response using JSON.parse() and demonstrated a basic example with error handling.

Changes made:
The explanation was used as a learning reference rather than copied directly. The actual AJAX implementations in quiz.js for loading questions.json, in main.js for the GitHub activity feed, and for the Dog CEO API reward were all written independently after understanding the concept. The error handling pattern: checking xhr.status === 200 and handling xhr.onerror separately was added based on thinking about what could go wrong in production.

Use case 3

Debugging assistance
GitHub Copilot

Prompt used:
My CSS stylesheet is not being applied to the page. The styles are not loading even though the link tag looks correct. What is the cause to this?
What the AI produced: GitHub Copilot suggested several possible code-related causes: a typo in the href attribute of the <link> tag, incorrect CSS selector syntax, specificity conflicts with Bootstrap overriding custom styles, or a missing closing brace in the CSS file causing rules to be ignored. It proceeded to suggest corrections to the link tag syntax and offered to rewrite several CSS rules.

Changes made:
None of the suggested code changes were applied because the problem was not a code issue at all. After checking the file system, the stylesheet had been saved in the project root folder instead of the css/ subfolder. The path href="css/style.css" was correct — the file was simply in the wrong location. Moving the file resolved the issue immediately. The AI assumed a code problem when the actual cause was environmental.

Critical Evaluation


Where the AI got it wrong

During development of the tutorial page, Claude was asked to generate the HTML structure for a tabbed content system with code snippet cards. The output looked correct when reading the code, but when rendered in the browser, content from one tab was bleeding into another tab's panel. After careful inspection, the cause was identified: a <pre> code block had been placed outside its parent <div class="card"> closing tag instead of inside it. 

<!-- What the AI generated -->
<div class="card">
  <h3>Topic</h3>
  <p>Explanation</p>
</div>
<pre><code>...snippet...</code></pre> <!-- outside the card -->

<!-- The correct syntax -->
<div class="card">
  <h3>Topic</h3>
  <p>Explanation</p>
  <pre><code>...snippet...</code></pre> <!-- inside the card -->
</div>

The error was very subtle. The code was syntactically valid HTML so no validator would catch it, and it looked reasonable when reading line by line. It was only detected by visually testing the page in the browser. This showed that AI-generated HTML can be structurally correct but semantically wrong in ways that only become visible at runtime.
